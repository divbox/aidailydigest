Banning Open Source AI Would Be A Mistake
Interconnects

Co-authored with Kevin Xu of Interconnected as a general-audience op-ed, this piece argues directly against the wave of restrictions now hitting open-source AI — from US Government policy to Anthropic's Fable 5 terms.

The core argument: The entire AI industry was built on open research. The transformer itself came from a freely published Google Brain paper. Restricting open source AI now — at the moment the technology is most powerful and most consequential — would cut off the very culture of collaborative research that got us here.

What triggered it: Anthropic's Claude Fable 5 launched with terms that explicitly bar developers from using it to train competing LLMs. The authors describe this as a "moment that, once seen, will be hard to unsee." Anthropic also initially implemented silent output degradation for users it detected doing LLM research — weakening model outputs without notifying the user — before walking back the policy after significant backlash.

The stakes: These restrictions aren't just philosophical. They're already causing enterprises and nation-states to aggressively build AI infrastructure they control outright — accelerating fragmentation and making reliable AI access a geopolitical concern rather than a commodity.

The analogy: The authors compare Anthropic's move to Microsoft's terms of use barring anyone from developing a competing OS — a restriction that would have been widely recognized as anticompetitive. The argument: safety justifications for these restrictions ring hollow when the same companies have benefited enormously from open access to foundational research.

Alpha Signal — Sunday, June 21
How Your Agents Can Write and Optimize Their Own Skills
Alpha Signal

The problem: Agent skill files (.md files that act as an agent's operating procedures) are still optimized manually — and it's painful. Fix a brittle behavior on Task A, cause a regression on Task B. Without systematic tracking, you can't even know which text change caused which failure. The underlying model gets gradient updates; the skill file gets a developer's guesses.

SkillOpt (Microsoft Research): Treats the skill document as a trainable external state — like a neural network parameter but for text. The pipeline runs a structured loop:

  • Rollout: Execute a batch of tasks, record trajectories.
  • Evaluate: Score trajectories as success or failure via a verifier.
  • Reflect: A separate LLM optimizer analyzes minibatches to identify which text components are driving failures.
  • Bounded edits: The optimizer proposes specific add/delete/replace modifications, constrained by a "textual learning-rate budget" to prevent volatile swings.

Proposed edits are validated on a held-out set not seen during training; failed edits go into a rejected-edit buffer. Results on GPT-5.5: +23.5 point average improvement in direct chat, +24.8 in a Codex loop, across 52 evaluated model/benchmark/harness settings. Final skill files average ~920 tokens. Skills optimized for one model/harness generalize to others, though in-distribution optimization is still preferable.

GEPA (Genetic-Pareto): Uses evolutionary algorithms to improve LLM instructions. After each task execution, an LLM reflects on the reasoning trace, diagnoses failures, and proposes "mutations" to the artifact. GEPA runs Pareto-based selection — maintaining a frontier of top candidates that each perform well across different tasks — and samples diverse winning strategies for further exploration. Compatible with DSPy.

EvoSkill: Applies GEPA's evolutionary approach specifically to multi-agent coding workflows. Keeps multiple skill candidates simultaneously on separate Git branches, uses a Pareto frontier to select the highest-performing variants, and evaluates branches on held-out datasets before promoting them.

The tradeoff: Automated skill optimization requires a verifiable feedback signal and a clean held-out eval set — it can't work on open-ended subjective tasks. Upfront compute cost is non-trivial (the LLM optimizer reads voluminous token histories). But the output is a standard text file with zero inference overhead, so optimization cost is one-time.

The bigger picture: This reflects a shift from linear prompting to "loop engineering" — assembling control systems with precise eval metrics, memory storage, and exit conditions, then letting agents run until they hit optimal performance. Production systems are already emerging that track live agent trajectories, flag recurring edge-case failures, and launch background optimization routines autonomously. The era of manually tweaking individual phrases in a skill file is closing.
